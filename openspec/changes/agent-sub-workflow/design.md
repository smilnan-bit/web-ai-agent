## Context

当前 Agent 工作流画布支持多种节点类型（LLM、代码、工具、条件分支等），但所有节点都局限在同一个画布内。工作流之间相互独立，无法相互引用。

本次变更在前端（工作流画布 + 列表）和后端（工作流数据模型 + 执行引擎）均有改动，属于跨层变更，需要提前明确技术决策。

**约束**：

- 支持最多 3 层嵌套（含主流程共 4 层深度）
- 子工作流始终按最新发布版本执行
- 不支持循环引用
- 本期仅做「引用」，不做「封装/解散」

## Goals / Non-Goals

**Goals:**

- 画布中新增 `workflow` 节点类型，可引用已发布的子工作流
- 子工作流节点的输入参数从引用工作流的 Start 节点读取（按最新发布版本）
- 子工作流节点的输出参数从引用工作流的 End 节点读取（按最新发布版本）
- 防止循环引用，限制嵌套深度 ≤ 3
- 工作流列表展示「工作流引用数」
- 删除工作流时若被引用则阻止删除
- 调试窗口支持查看子工作流节点的输入/输出日志
- 「条件分支」节点重命名为「条件判断」

**Non-Goals:**

- 封装/解散子流程（P1，本期不做）
- 查看工作流引用关系图（P1，本期不做）
- 子工作流版本锁定（始终跟随最新发布版本，不支持指定版本号）
- 子工作流的权限隔离（不在本期考虑）

## Decisions

### D1：工作流节点的数据结构

**决策**：在工作流节点 DSL 中新增 `type: "workflow"` 的节点类型，节点数据包含：

```ts
interface WorkflowNodeData {
  type: 'workflow';
  workflowId: string; // 引用的子工作流 ID
  inputs: ParamMapping[]; // 输入参数映射（变量名 → 引用/固定值）
  errorHandling: {
    strategy: 'abort' | 'return' | 'branch';
    returnValue?: Record<string, unknown>; // strategy = "return" 时使用
  };
}

interface ParamMapping {
  name: string; // 参数名（来自子工作流 Start 节点，只读）
  type: string; // 参数类型（只读）
  value: {
    mode: 'ref' | 'literal';
    ref?: string; // 前序节点输出的引用路径
    literal?: unknown; // 固定值
  };
}
```

**理由**：复用现有节点 DSL 模式，最小化画布渲染层改动。

---

### D2：子工作流参数始终读取最新发布版本

**决策**：表单每次展开时，通过 API 实时拉取子工作流的最新发布版本的 Start/End 节点参数，不在节点 DSL 中缓存参数快照。

**理由**：子工作流更新后，引用方无需手动同步参数定义，始终保持一致。

**代价**：表单展开时有一次网络请求，需要加 loading 态。

---

### D3：循环引用检测策略

**决策**：在「选择工作流」弹窗的列表接口侧进行过滤（服务端），前端不做额外校验。

过滤逻辑（后端）：

1. 给定当前工作流 `A`，获取所有工作流 `W`
2. 对每个 `W`，递归检查其引用链中是否包含 `A`（DFS）
3. 过滤掉包含 `A` 的所有 `W`
4. 同时过滤掉嵌套深度 `depth(W) + 1 ≥ 4` 的工作流

**理由**：集中在服务端处理，避免前端维护引用关系图，逻辑更可靠。

---

### D4：嵌套层级限制实现

**决策**：每个工作流在发布/保存时计算并存储 `maxDepth` 字段（最大嵌套深度）：

- 叶子工作流（无子工作流节点）：`maxDepth = 1`
- 含子工作流节点的工作流：`maxDepth = max(子工作流的 maxDepth) + 1`

选择弹窗接口过滤条件：`currentWorkflow.maxDepth + candidate.maxDepth < 4`

**理由**：避免每次弹窗都做全量深度遍历，性能更好。

---

### D5：异常处理与「执行异常流程」分支桩

**决策**：当异常处理策略为 `"branch"` 时，工作流节点卡片底部展示额外的「异常」连接桩，行为与「条件分支」节点一致（复用现有异常分支连接桩逻辑）。

**理由**：复用已有实现，减少新增代码量。

---

### D6：「条件分支」→「条件判断」重命名

**决策**：仅修改节点的显示名称（`label`），不修改节点的 `type` 字段，以保证现有数据兼容。

**理由**：存量工作流中已有大量 `type: "condition"` 节点，修改 type 会导致数据迁移问题。

---

### D7：工作流引用数统计

**决策**：后端在工作流列表接口中直接返回 `refCount`（被几个工作流引用），而不是由前端实时查询。

`refCount` 在以下时机更新：

- 工作流节点被添加/删除时
- 工作流被删除时

**理由**：列表页按需展示，不适合前端批量查询，服务端统计更高效。

## Risks / Trade-offs

| 风险                                                   | 缓解措施                                                                       |
| ------------------------------------------------------ | ------------------------------------------------------------------------------ |
| 子工作流版本更新导致参数不兼容（如删除了原有输出参数） | 发布校验时检查输入映射的引用参数是否仍存在于最新版本中，不存在则阻止发布并提示 |
| 嵌套执行超时/无限递归（若 maxDepth 计算有 bug）        | 执行引擎增加运行时嵌套计数器，超过 4 层强制中断并返回错误                      |
| 选择工作流弹窗列表接口性能（大量工作流时 DFS 耗时）    | 增加 `maxDepth` 字段缓存，避免每次全量递归；必要时加缓存层                     |
| 调试日志量大（子工作流层层嵌套）                       | 调试窗口按节点折叠展示，默认收起子工作流日志                                   |
| 删除保护弹窗中引用方数量过多                           | 最多展示前 10 条，超出显示"等 N 个工作流"                                      |

## Migration Plan

1. **后端**：

   - 新增工作流节点类型支持（DSL schema 变更，向后兼容）
   - 新增接口：`GET /workflows?excludeIds=&maxDepth=` (选择弹窗)
   - 新增接口：`GET /workflows/{id}/params?version=latest` (节点表单)
   - 新增字段：工作流表 `max_depth INT`, `ref_count INT`
   - 已有工作流数据无需迁移（新字段默认值为 0/1）

2. **前端**：

   - 新增 `WorkflowNode` 组件（卡片 + 连接桩）
   - 新增 `WorkflowNodeForm` 右侧面板组件
   - 新增 `WorkflowSelectionModal` 弹窗
   - 修改节点类型列表（顶部插入 workflow，条件分支改名）
   - 修改工作流列表页（新增引用数列）
   - 修改删除工作流弹窗（引用检查）

3. **回滚**：新节点类型为纯扩展，回滚时关闭入口即可；已保存的含 `workflow` 节点的工作流需要清理

## Open Questions

1. **子工作流参数变更的提示**：当子工作流更新后，引用方表单中的参数映射是否需要在打开时提示"参数已变更，请重新配置"？还是静默更新？
2. **老工作流的判断标准**：「过滤老工作流」的判断字段是什么？通过 `version` 字段区分，还是有专门的 `isLegacy` 标记？
3. **调试日志格式**：子工作流节点的日志是嵌套展示（树形），还是平铺在主流程日志序列中？
4. **并发引用写入**：多用户同时修改引用关系时 `ref_count` 的一致性如何保证（乐观锁 or 原子更新）？
