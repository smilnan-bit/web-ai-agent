## Why

当前 Agent 工作流仅支持在单一画布内构建全量逻辑，无法跨团队协作维护或复用已有工作流。客户（孩子王）拥有 20+ 工作流，需要多名员工分别维护各自的子流程后，合并到一个主流程中统一运行，因此需要引入「引用子工作流」能力。

## What Changes

- 新增「工作流节点」，允许在画布中引用一个已存在的工作流作为子流程执行
- 将原「条件分支」节点重命名为「条件判断」
- 新增「选择工作流」弹窗，支持搜索和添加子流程（带循环引用/嵌套深度/老工作流过滤）
- 工作流节点表单支持配置：子工作流信息（只读）、输入参数映射、输出参数展示、异常处理
- 工作流节点卡片展示子流程名称、描述、输入/输出摘要，支持复制与重命名
- 工作流发布时对工作流节点进行合法性校验
- Agent 调试窗口支持查看子工作流节点的输入/输出日志
- 工作流列表新增「工作流引用数」字段（被几个工作流引用）
- 删除工作流时，若被引用则弹窗阻止删除并列出引用方列表
- 子工作流执行时始终使用其最新发布版本

## Capabilities

### New Capabilities

- `workflow-node`: 画布中新增"工作流节点"类型，支持引用子工作流并配置输入/输出/异常处理
- `workflow-selection-modal`: 选择子工作流的弹窗，子工作流列表从接口获取。子工作流弹窗弹出的逻辑复用工具节点添加工具弹窗弹出逻辑
- `workflow-node-form`: 工作流节点右侧表单，展示子工作流信息、输入参数映射、输出参数、异常处理配置
- `workflow-node-card`: 画布中工作流节点卡片的 UI 规格（名称/描述/参数摘要/异常分支桩）
- `workflow-node-validation`: 工作流发布前对工作流节点的校验规则
- `sub-workflow-debug-log`: Agent 调试窗口中展示子工作流节点的运行日志
- `workflow-reference-count`: 工作流列表中新增「工作流引用数」列，展示被引用次数
- `workflow-delete-guard`: 删除工作流时的引用检查与阻断弹窗
- `sub-workflow-version-policy`: 子工作流执行时始终使用最新发布版本的策略规则

### Modified Capabilities

- `workflow-node-types`: 「条件分支」节点重命名为「条件判断」，且节点列表顶部新增「工作流」节点入口

## Impact

- **前端**：工作流画布（节点列表、节点卡片、连接桩）、节点表单面板、发布校验逻辑、调试日志面板、工作流列表页
- **后端/API**：需要新接口支持：工作流列表（带引用过滤）、工作流节点输入/输出参数获取（按发布版本）、引用关系查询、删除前引用校验
- **数据模型**：工作流节点新增 `type: workflow`，需存储引用的 `workflowId`、输入参数映射、异常处理策略
- **执行引擎**：子工作流节点执行时需调用子工作流的最新发布版本，支持最多 3 层嵌套
