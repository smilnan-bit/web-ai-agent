# Agent 对接 JSON-Render 访客端技术方案

> 关联 OpenSpec change: `agent-cotui-visitor-integration`
> 覆盖仓库：`web-ai-agent`（管理后台） + `basic-web-visitor`（访客端）
> 版本：V3（独立 cmd 方案）

---

## 1. 背景

CardEditor（`web-ai-agent` 内 `/card/editor`）已具备将 LLM 产出的 `PreviewSpec`（json-render 风格卡片）保存到后端的能力，但：

- 访客端无法渲染 CotUi 卡片
- 工作流 Dialog 节点只能下发 product/order 两种预定义卡片，不支持引用 CotUi spec

本方案补齐「LLM 生成结构化卡片 → 工作流编排 → 访客端渲染 → 用户交互回传」的端到端链路。

## 2. 整体架构

```
┌──────────────────────────┐         ┌────────────────────────────┐
│  web-ai-agent            │         │  basic-web-visitor         │
│  （管理后台）            │         │  （访客端）                │
│                          │         │                            │
│  CardEditor              │         │  IM 层 cmd 分发            │
│    ↓ 产出 PreviewSpec    │         │    ↓ cmd 13004             │
│  /cotui/spec/save        │         │  QyMessage(COTUI)          │
│                          │         │    ↓                       │
│  Workflow Editor         │         │  msgConfigMap[COTUI]       │
│  Dialog.cotUi 配置       │         │    ↓                       │
│    ↓ specId + bindings   │         │  CotUiMsg                  │
└───────┬──────────────────┘         │    ↓                       │
        │                            │  Renderer (SpecPreview)    │
        ▼                            │    ↓ 用户点击 submit        │
┌──────────────────────────┐         │  emit.ts → cmd 13005 ──────┼──┐
│  后端（不在本次前端范围）│         └────────────────────────────┘  │
│                          │                                         │
│  1. 工作流执行 Dialog    │◀────────────────────────────────────────┘
│  2. 解析 binding + runtimeData                                     │
│  3. 发 cmd 13004 (msgidClient, cardId, spec, runtimeData)          │
│  4. 内部以 msgidClient 为主键存工作流节点映射                      │
│  5. 收 cmd 13005 → 按 msgidClient 查表 → actionName 路由 →         │
│     USER_RESPONSE                                                  │
│  6. 超时独立计时：askUserLimit 到期 → otherwise 分支                │
│  7. 工作流继续                                                     │
└──────────────────────────┘
```

## 3. 协议契约

### 3.1 下发：`cmd 13004`（server → visitor）

文件：`source/protocol/cmd_13004.ts`

| 字段            | 类型                  | 必填 | 说明                                                             |
| --------------- | --------------------- | ---- | ---------------------------------------------------------------- |
| `cmd`           | `13004`               | ✅   | 固定值                                                           |
| `sessionid`     | number                | ✅   | 会话 id（IM 协议通用）                                           |
| `msgidClient`   | string                | ✅   | 幂等 key；**同时作为回源索引**，后端以它为主键维护工作流节点映射 |
| `cardId`        | string \| number      | ✅   | 卡片身份标识，来自后端 `/cotui/spec/save` 返回的 id              |
| `spec`          | PreviewSpec           | ✅   | 完整卡片描述，**永远内联下发**                                   |
| `runtimeData`   | `Record<string, any>` | ⭕   | 后端 resolve binding 后的数据快照                                |
| `fallback`      | `{ text: string }`    | ✅   | 访客端解析失败时的降级文本                                       |

> ⚠️ 已裁剪字段：
>
> - `mode` / `specId` —— spec 永远内联，大小由后端在保存接口层约束（如 32KB）
> - `schemaVersion` —— 卡片结构由 `cardId` 对应的后端记录决定；版本演进由后端在下发前转换，访客端无需感知
> - `outputStyle` —— 卡片展示形态由访客端自行决定
> - `timeoutMs` —— 超时由后端独立处理，访客端不感知
> - `dialogId / flowId / nodeId / nodeType / requestId / preRequestId` —— 回源由后端按 `msgidClient` 查表还原

### 3.2 上报：`cmd 13005`（visitor → server）

文件：`source/protocol/cmd_13005.ts`

| 字段          | 类型         | 必填 | 说明                                        |
| ------------- | ------------ | ---- | ------------------------------------------- |
| `cmd`         | `13005`      | ✅   | 固定值                                      |
| `sessionid`   | number       | ✅   | 会话 id                                     |
| `msgidClient` | string       | ✅   | 对应下发消息 id；后端据此查表定位工作流节点 |
| `event`       | `CotUiEvent` | ✅   | 见下                                        |

> ⚠️ **不**携带任何工作流上下文字段，仅靠 `msgidClient` 让后端还原回源。

`CotUiEvent`：

```typescript
interface CotUiEvent {
  type: 'action' | 'lifecycle';
  actionType?: 'emit' | 'navigate' | 'submit'; // type=action 必填
  actionName?: string; // type=action 必填
  buttonNodeId?: string;
  staticPayload?: Record<string, any>;
  formValues?: Record<string, string | string[]>;
  lifecycle?: 'render' | 'close'; // type=lifecycle 必填（仅埋点可选）
  timestamp: number;
}
```

> ⚠️ `lifecycle` 去掉了 `'timeout'` —— 超时由后端独立计时，访客端不感知。

### 3.3 USER_RESPONSE 映射

后端收到 cmd 13005 后，按 `event.actionName` 路由分支，USER_RESPONSE 结构：

```typescript
USER_RESPONSE = {
  actionName: string,
  actionType: 'emit' | 'navigate' | 'submit',
  // CardConfig.cotUi.outputKeys 选中的字段被展平
  username: string,
  size: string,
  colors: string[],
  // staticPayload 同样被展平
  sku: string,
}
```

## 4. 按钮状态机

```
                     ┌────────────┐
                     │  rendered  │
                     └─────┬──────┘
     ┌───────────────┬─────┼─────┬───────────────┐
     │ submit        │ emit│     │ navigate      │
     ▼               ▼     │     ▼               │
 ┌─────────┐    ┌─────────┐│  ┌─────────┐
 │submitted│    │ rendered││  │ rendered│
 │ (全灰)  │    │ (可再点)││  │ +openURL│
 └─────────┘    └─────────┘│  └─────────┘
      │                    │        │
      └────────────────────┴────────┘
                           │
                           ▼
                  访客端发 cmd 13005（navigate 除外）
```

> 超时**不**由访客端感知。后端 `askUserLimit` 到期后主动走 otherwise 分支，访客端卡片按钮不会自动置灰——用户仍可点击，但 cmd 13005 到达后端时会因节点已终态而被静默丢弃。

| action.type | 回源 | 卡片状态        | 允许重复 |
| ----------- | ---- | --------------- | -------- |
| `submit`    | ✅   | 整卡置灰        | ❌       |
| `emit`      | ✅   | 不变            | ✅       |
| `navigate`  | ❌   | 不变 + 打开 URL | ✅       |

## 6. 关键时态决策

### 6.1 IM 消息体存什么

访客端 IM 消息体 = `spec` + `runtimeData` + 回源字段 + 本地状态（`submitted / expired`）。

**原因**：

- 回看需要稳定快照，后续 spec 改版不影响历史
- runtimeData 当时变量值需保留（通常不含敏感字段，若含由后端脱敏）
- `submitted` 状态本地持久化，防止用户回看时看到可再次点击的已提交卡片

### 6.2 何时置灰

```
  用户点 submit → 访客端立即本地置灰
       ↓
  发 cmd 13005
       ↓
  （后端 ACK / 下一条消息到达）→ 最终确认
       ↓
  失败回滚？→ 本期不做，保持"先置灰"
```

## 8. 边界与兼容

| 场景                     | 策略                                                                                                                  |
| ------------------------ | --------------------------------------------------------------------------------------------------------------------- |
| 旧版访客端收到 cmd 13004 | IM 层丢弃；后端按 SDK 版本**主动降级**为 cmd 203 + radio_button                                                       |
| spec 超大                | 后端 `/cotui/spec/save` 接口做字节数上限校验（建议 32KB），超限保存失败，不进入下发链路                               |
| 重复下发 / 断网重连      | `msgidClient` 幂等                                                                                                    |
| 用户连点 submit          | 本地立即 disable 整卡按钮                                                                                             |
| 超时                     | 访客端不感知；后端独立计时，到期走 otherwise；访客端卡片按钮仍可点，后端用 `msgidClient` 识别已终态节点并静默丢弃事件 |
| XSS                      | text 仅 textContent；image.src 白名单；action.type 枚举外忽略                                                         |
| 历史回看 spec 改版       | 消息体内快照保证稳定                                                                                                  |
| 多卡片共存               | 以 `msgidClient` 区分回源（后端按 msgidClient 查工作流节点）                                                          |

## 附录 A：协议演进空间

- 本次仅支持全量 spec 下发。**未来**若接入流式生成，可在同一 cmd 13004 内新增 `streaming: true` + `chunkId` 字段，或新开 cmd 13006 承载增量 patch。
- 本次 CotUi 节点类型冻结为 7 种。**未来**扩展时由后端按访客端 SDK 版本裁剪 spec 后再下发（访客端收到什么就渲染什么，不在协议层感知版本）。
- 本次 `navigate` 动作不回源。**未来**若要统计点击率，可单开 lifecycle 事件 `navigate_clicked`，不改变分支路由语义。
